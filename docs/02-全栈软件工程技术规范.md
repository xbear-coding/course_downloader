# 全栈软件工程技术规范（示例：Course_Downloader）

> 版本：v1.0 | 日期：2026-05-14
> 本文档以 Course_Downloader 项目为实例，展示全栈软件开发全过程的标准化规范化实践。
> 涵盖：需求分析 → 架构设计 → 技术选型 → 数据库设计 → API 设计 → 错误处理 → 安全规范 → 实施流程。

---

## 第一章：全栈开发总纲

### 1.1 开发生命周期

```
需求澄清 ──→ 架构设计 ──→ 任务拆解 ──→ 增量实现 ──→ 验证交付
    │            │            │            │            │
    ▼            ▼            ▼            ▼            ▼
  Brainstorm  多方案对比   原子化任务   每次 ~100行   验收标准
  用户确认   ADR决策     检查点        TDD优先     证据门禁
```

### 1.2 规范标准化清单

| 阶段 | 规范项 | 本项目的实践 | 避坑提示 |
|------|--------|-------------|---------|
| 需求 | 先问「为什么」再问「怎么做」 | 腾讯会议+小鹅通合并为一个工具的原因：核心流程相同 | 不要用户说做什么就做什么，先理解他想解决什么问题 |
| 需求 | 列出假设清单让用户确认 | 「我假设你是个人免费版账号」等 12 条假设 | 不列假设 → 做到一半发现前提错了 |
| 需求 | 区分「必须做」和「可以不做」 | 抖音/小红书有技术风险，标注可行性验证 | 不要把高风险项放在最后才发现 |
| 设计 | 每个关键决策至少 2 个备选 | ASR 方案对比 6 个平台最后选 SiliconFlow | 只有一个方案的决策 ≈ 没做决策 |
| 设计 | ADR 记录决策理由 | FastAPI 选型理由：异步、自动文档、WebSocket | 不记录理由 → 半年后没人知道为什么这么选 |
| 接口 | 接口先定义再实现 | BasePlatform + Mixin 接口在 Phase 1 定义，Phase 2 才实现 | 接口没稳定就实现 → 后面接口改了实现全废 |
| 实现 | 增量切片，每次 ~100 行 | 20 个原子任务，每个 S/M 级 | 一次改 1000 行的 PR → 审查不了、回滚不了 |
| 实现 | TDD 优先 | 后端用 pytest + pytest-asyncio | 没有测试的代码 = 不可信的代码 |
| 安全 | Always Do 列表 | Phase 1 必须做：CORS、global handler、.gitignore | 安全后补 = 成本 ×10 |
| 验证 | 证据门禁 | 每个 task 有验证命令 + 预期输出 |「看起来对了」不算完成 |

## 第二章：需求澄清规范

### 2.1 需求澄清检查清单

拿到一个需求后，按顺序问自己这 5 个问题：

```
1. 用户想解决什么问题？
   → 不是「做个下载工具」，是「不想每次手动点导出」
   → 追问：现在是手动点的哪几步？每一步多久？

2. 不做这个需求有什么影响？
   → 每天花 15 分钟手工操作
   → 影响量化后才能判断优先级

3. 这个需求是不是另一个需求的子集？
   → 小鹅通和腾讯会议都是「浏览器登录→下载内容」
   → 本质上是一个需求的两种实例

4. 最简可行方案是什么？
   → 只做腾讯会议，不做 6 平台
   → 但用户说一次全上—尊重用户决定

5. 哪里可能出错？
   → 抖音/小红书反爬
   → ASR 时间戳不支持
   → 每个风险点都要标注可行性验证
```

### 2.2 需求文档结构

```markdown
# 需求：[功能名称]

## 用户故事
作为一个[角色]，我想要[功能]，以便[价值]。

## 验收标准（可测试）
- [ ] 标准 1：...
- [ ] 标准 2：...

## 假设
- 假设 1：...
- 假设 2：...

## 不做的范围
- 不包含：...
```

### 2.3 需求阶段避坑提示

| 坑 | 表现 | 预防 |
|----|------|------|
| **假设沉默** | 用户没说 = 你以为不存在 | 主动列出所有假设让用户确认 |
| **范围蔓延** |「顺便加个导出 Excel」 | 区分「必须/应该/可以」 |
| **技术不可行** | 抖音反爬走不通 | 高风险项先做可行性验证 |
| **用户以为 A 其实是 B** |「多用户」= 多腾讯会议账号，不是多系统用户 | 歧义术语追问澄清 |
| **过度承诺** | 6 平台一期全上 | 标注每个平台的风险等级 |

## 第三章：架构设计规范

### 3.1 技术选型决策矩阵

每个关键技术选择必须有对比表：

```
| 方案 | 优势 | 劣势 | 选？ | 理由 |
|------|------|------|------|------|
| Flask | 简单、生态大 | 同步、无原生WebSocket | ❌ | 需要异步+WS |
| FastAPI | 异步、自动文档、WS | 较新、生态略小 | ✅ | 符合需求 |
```

**选型原则：**
1. 优先选团队熟悉的技术（个人项目 = 你熟悉的）
2. 不追新（Playwright 不是最新版，是 1.58 LTS）
3. 不选大炮打蚊子（个人工具用 PostgreSQL 就是）
4. 验证通过再确认（SiliconFlow Key 先测通再写进架构）

### 3.2 分层架构规范

```
表示层（Vue SPA）
    ↑ HTTP/WebSocket
接口层（FastAPI）
    ↓
服务层（业务逻辑）
    ↓
数据层（SQLite）
```

**分层原则：**
- 上层依赖下层，下层不依赖上层
- 同层之间通过接口通信，不直接调用
- 每层只做自己的事：路由层不写业务逻辑，服务层不处理 HTTP

### 3.3 插件架构设计

```python
class BasePlatform(ABC):
    """所有平台必须实现"""
    @abstractmethod
    def login(self) -> LoginResult: ...
    @abstractmethod
    def fetch_list(self, page_token: Optional[str] = None) -> FetchResult: ...
    @abstractmethod
    def get_metadata(self, item: ContentItem) -> dict: ...

class VideoCapable(ABC):
    @abstractmethod
    def download_video(self, item: ContentItem, output: Path) -> DownloadResult: ...
```

**插件设计避坑：**
| 坑 | 表现 | 预防 |
|----|------|------|
| 接口太多 | 每个平台都要实现不用的方法 | 拆 Mixin |
| 返回值太简单 | bool 无法表达失败原因 | 用 Result 数据类 |
| 缺少分页 | 一次返回所有数据 | 加游标/页码参数 |

### 3.4 架构图规范

每份架构文档必须包含：
1. **分层架构图**（四层模型）
2. **数据流图**（数据从哪里来到哪里去）
3. **ER 图**（数据库表关系）
4. **时序图**（关键流程的交互顺序）

图例工具：HTML SVG（`architecture-diagram-creator` skill）

## 第四章：API 设计规范

### 4.1 RESTful 设计标准

```
✅ 正确                      ❌ 错误
GET    /api/tasks            GET    /api/getTaskList
POST   /api/tasks            POST   /api/createTask
GET    /api/tasks/{id}       GET    /api/task?id=123
PATCH  /api/tasks/{id}       POST   /api/updateTask
DELETE /api/tasks/{id}       GET    /api/deleteTask?id=123
```

**命名规范：**
- 资源名用**复数名词**：`tasks` 不是 `task`
- URL 路径用 **kebab-case**：`/api/api-keys` 不是 `/api/api_keys`
- 查询参数用 **camelCase**：`?sortBy=createdAt`
- 不要用动词：不用 `/api/getTasks`，用 `GET /api/tasks`

### 4.2 统一错误响应

```json
// 成功
{"data": {...}, "pagination": {"page": 1, "pageSize": 20, "total": 100}}

// 错误
{"error": {"code": "VALIDATION_ERROR", "message": "标题不能为空", "detail": {...}}}
```

**HTTP 状态码：**
| 码 | 含义 | 使用场景 |
|----|------|---------|
| 200 | 成功 | GET、PATCH |
| 201 | 创建成功 | POST |
| 400 | 请求参数错误 | 校验失败 |
| 401 | 未认证 | 需要登录 |
| 404 | 资源不存在 | ID 不对 |
| 409 | 冲突 | 重复创建 |
| 422 | 语义错误 | Pydantic 校验失败 |
| 500 | 服务端错误 | 不暴露详情 |

### 4.3 分页规范

```python
# 请求
GET /api/tasks?page=1&pageSize=20&sortBy=createdAt&sortOrder=desc

# 响应
{"data": [...], "pagination": {"page": 1, "pageSize": 20, "totalItems": 142, "totalPages": 8}}
```

**所有列表端点必须加**分页，即使当前数据量小——当用户积累几千条任务时，没分页的接口直接炸。

## 第五章：数据库设计规范

### 5.1 字段规范

```python
# ✅ 正确
class Task(Base):
    __tablename__ = "tasks"  # 复数 snake_case
    id = Column(Integer, primary_key=True)
    resource_id = Column(String(100), unique=True)  # 唯一约束
    status = Column(Enum(TaskStatus))               # 枚举类型
    created_at = Column(DateTime, default=datetime.utcnow)

# ❌ 错误
class task(Base):           # 不应该用小写单数
    task_id = Column(...)   # 不要用 task_id，用 id
    status = Column(String) # 不要用 TEXT，会写入脏数据
```

### 5.2 命名规范

| 对象 | 规范 | 示例 |
|------|------|------|
| 表名 | 复数 snake_case | `api_keys`，`download_tasks` |
| 主键 | `id` | 不要用 `task_id` |
| 外键 | `{表名}_id` | `account_id` |
| 时间字段 | `created_at`，`updated_at` | 统一命名 |
| 布尔字段 | `is_` 前缀 | `is_active`，`is_paused` |

### 5.3 索引策略

```python
# 经常按状态查询 → 加索引
Index("idx_tasks_status", "status")

# 经常按平台 + 状态查 → 联合索引
Index("idx_tasks_platform_status", "platform", "status")

# 外键自动加索引（SQLAlchemy 默认）
```

**避坑：**
- 不要给所有字段加索引（写入变慢）
- 不要不加任何索引（查询变慢）
- 个人工具数据量小（< 10 万条），索引不是关键问题

## 第六章：错误处理规范

### 6.1 三层兜底模式

```
第 1 层：操作级别
  操作失败 → 3 次重试（0s → 5s → 15s）
  全部失败 → 进入第 2 层

第 2 层：任务级别
  子步骤失败 → 标记独立状态，不影响其他步骤
  任务结束时对失败步骤整体重试
  仍失败 → 进入第 3 层

第 3 层：系统级别
  标记 fatal → 不再自动重试
  UI 提供「重试」和「忽略」按钮
  记录完整错误日志
```

### 6.2 错误记录规范

```python
import logging

logger = logging.getLogger(__name__)

try:
    result = download_video(url)
except TimeoutError as e:
    logger.error(f"[{task.id}] 视频下载超时: {url[:50]}...")
    task.video_status = "failed"
    task.error_message = f"TIMEOUT: {str(e)[:100]}"
except Exception as e:
    logger.exception(f"[{task.id}] 视频下载异常")  # 记录完整 traceback
    task.video_status = "failed"
    task.error_message = f"UNEXPECTED: {str(e)[:100]}"
```

### 6.3 监控指标

```
WebSocket 每 1 秒推送进度
5 分钟无进度 → 触发第 2 层重试
10 分钟无响应 → 触发第 3 层重置
任务完成后发送摘要（成功/失败/跳过数量）
```

## 第七章：安全规范

### 7.1 分级安全策略

```
▸ Always Do（必须做）
  - CORS 配置（不配 → Vue 连不上）
  - 全局异常处理（不配 → stack trace 暴露给用户）
  - .gitignore（不配 → API Key 被提交到 git）
  - 日志脱敏（API Key 只显示后 4 位）
  - 数据库参数化查询（SQLAlchemy ORM 默认）

▸ 个人工具不需要做
  - API Key 加密存储（加密密钥没地方放）
  - CSP / HSTS / HTTPS（本地 HTTP 够了）
  - 用户认证系统（不需要登录）
```

### 7.2 敏感数据处理

```python
# API Key 存储：明文 SQLite + 日志脱敏
class APIKey(Base):
    key_value = Column(String(500), nullable=False)  # 明文

# 日志脱敏
def mask_key(key: str) -> str:
    if len(key) < 12:
        return "***"
    return key[:4] + "..." + key[-4:]

logger.info(f"Testing key: {mask_key(key_value)}")
# 输出：sk-q...hsrh
```

### 7.3 XSS 防护

```html
<!-- ✅ 安全：Vue 默认转义 -->
<div>{{ article.title }}</div>

<!-- ❌ 危险：直接渲染 HTML -->
<div v-html="article.content"></div>

<!-- ✅ 必须用 v-html 时，先清洗 -->
<div v-html="DOMPurify.sanitize(article.content)"></div>
```

## 第八章：测试规范

### 8.1 测试金字塔

```
        ╱╲
       ╱  ╲          E2E（5%）
      ╱──────╲
     ╱        ╲      集成（15%）
    ╱────────────╲
   ╱              ╲   单元（80%）
  ╱──────────────────╲
```

| 层 | 测试内容 | 框架 | 本项目做吗？ |
|----|---------|------|------------|
| 单元 | Service 层逻辑 | pytest | ✅ 做（关键路径） |
| 集成 | 数据库读写 | pytest + SQLite | ✅ 做 |
| 集成 | API 端点 | TestClient | ✅ 做 |
| E2E | 用户关键路径 | Playwright | ❌ 不做（个人工具） |

### 8.2 测试命名

```python
# ✅ 描述性命名
def test_fetch_list_returns_courses_when_logged_in():
def test_fetch_list_returns_empty_when_cookie_expired():
def test_download_video_retries_on_timeout():

# ❌ 无意义命名
def test_fetch():
def test_download():
```

### 8.3 不测试的内容

```
不测试 Playwright 对腾讯会议/小鹅通的浏览器操作（需要真实登录态）
不测试第三方 API（SiliconFlow、B站 API）—— mock 即可
不测试前端 UI（个人工具，ROI 低）
```

## 第九章：实施流程规范

### 9.1 任务拆解标准

| 粒度 | 文件数 | 适合场景 |
|------|--------|---------|
| XS | 1 | 修改配置、加字段 |
| S | 1-2 | 加单个 API、单个组件 |
| M | 3-5 | 完整功能切片 |
| L | 5-8 | **需要拆分** |

**每个任务必须包含：**
- 精确文件路径
- 验收标准（可测试）
- 验证命令 + 预期输出
- 依赖关系（前一个任务是什么）

### 9.2 检查点设置

```
每 2-3 个 Task 设置一个检查点
检查点必须：
  - 有可运行的验证命令
  - 有明确的通过/不通过标准
  - 通过后才能进入下一组 Task
```

### 9.3 增量实现原则

```
1. 每个变更控制在 ~100 行（超过 300 行必须拆分）
2. 每个切片必须：实现 → 测试 → 验证 → 提交
3. 重构和功能变更分开提交
4. 每个切片提交后系统保持可工作状态
```

## 第十章：全程避坑清单

### 10.1 需求阶段

| 坑 | 表现 | 预防 |
|----|------|------|
| 假设沉默 | 用户没说 = 你以为不存在 | 主动列出假设让用户确认 |
| 范围蔓延 |「顺便加个导出 Excel」 | 区分必须/应该/可以 |
| 术语歧义 |「多用户」含义不清 | 追问具体场景 |
| 技术不可行 | 抖音反爬走不通 | 高风险项先做可行性验证 |
| 用户不知道自己要什么 |「你看着做」 | 给 2-3 个方案让用户选 |

### 10.2 架构阶段

| 坑 | 表现 | 预防 |
|----|------|------|
| 选型没对比 | 直接选了一个 | 至少列 2 个方案对比 |
| 过度设计 | 个人工具用 PostgreSQL | 按场景选型 |
| 低估工作量 | Playwright 迁移当「改个库名」| 每个迁移都有兼容成本 |
| 忽视错误处理 | 只设计成功路径 | 每个环节分析故障模式 |

### 10.3 编码阶段

| 坑 | 表现 | 预防 |
|----|------|------|
| 接口返回值太简单 | `bool` 无法表达失败原因 | 用 Result 数据类 |
| 静默失败 | 返回空列表不报错 | 每个外部调用检查结果 |
| 文件命名不一致 | 同一个项目 snake_case + camelCase 混用 | 统一规范 |
| 没有 .gitignore | 密钥被提交 | Phase 1 就创建 |
| 日志没脱敏 | API Key 明文打印 | 统一 mask 函数 |

### 10.4 测试阶段

| 坑 | 表现 | 预防 |
|----|------|------|
| 测试太晚 | 写完代码才写测试 | Phase 1 就搭好测试框架 |
| mock 过度 | mock 了数据库+网络+时间 | 只 mock 外部依赖，内部用真实的 |
| 测试依赖顺序 | 测试 B 依赖测试 A 的数据 | 每个测试独立 setup/teardown |

### 10.5 隐藏技巧清单

| 技巧 | 说明 | 场景 |
|------|------|------|
| **可行性验证冲刺** | 高风险项先花 1 天验证 | 抖音反爬、Playwright 兼容性 |
| **技术选型的 3 天法则** | 新技术评估不超过 3 天 | Playwright vs Selenium |
| **ADR 的「理由」比「结论」重要** | 记下为什么选这个 | 半年后回看知道原因 |
| **接口的「最少承诺原则」** | 只暴露调用方需要的 | 避免 Hyrum's Law |
| **日志的「写完就删」原则** | 调试日志完成就删 | 避免日志膨胀 |
| **TODO 标注责任人** | `# TODO(@user): reason` | vs `# TODO: fix this` — 后者永远不修 |
| **Makefile 是活的入口文档** | `make install` `make dev` `make test` | 新人（包括未来的你）第一眼就看这 |
| **.env.example 必须提交** | 模板变量 vs 真实密钥 | 新人知道要配什么 |
