Course_Downloader — 业务交互流程图

📋 用户操作全流程

用户流程的关键设计：扫描更新和直接下载是两条并行的路径，用户可以主动触发扫描，也可以直接选课程下载。WebSocket 分步骤推送让用户知道当前进度到哪一步了。

🎛 下载任务生命周期状态机

状态说明

  • NEW — 扫描发现的新课程，等待用户确认
  • UPDATED — 已下载课程但服务端有更新，等待用户确认重下
  • PENDING — 已确认下载，在队列中等待
  • RUNNING — 正在处理
  • PARTIAL — 部分成功（视频下了但 ASR 失败）
  • DONE — 全部成功
  • FAILED — 全部失败（可重试）
  • FATAL — 重试超过上限，不再自动重试

子步骤状态（独立）

  • video_status — 视频下载
  • transcript_status — ASR 转写
  • subtitle_status — 字幕下载（B站）
  • article_status — 文章抓取（头条/小红书）
  • 每个子步骤独立记录 pending / running / done / failed / skipped

🔁 三层错误恢复流程

⏱ 实时监测

  • WebSocket 每 1 秒推送子步骤状态
  • 子步骤 5 分钟无进度 → 自动 kill 重启
  • 浏览器 10 分钟无响应 → 关闭 + 重启

🔄 自动纠错

  • 下载连接 30 秒无数据 → 断线重连
  • ASR API 60 秒无响应 → 取消 + 重新提交
  • 崩溃重启 → SQLite WAL 模式防损坏

🛑 自动重置

  • 浏览器崩溃 → 自动启动新实例
  • 数据库连接丢失 → 自动重连
  • 超过 3 次恢复仍失败 → 标记 FATAL

📊 增量检测流程

增量检测的核心：用 resource_id（而非标题）做唯一标识，用服务器更新时间戳和视频时长变化判断是否有更新，避免重复下载和漏下载。

🎤 ASR 转写管道

输出格式示例

# 课程名称 - 转写稿

**00:00:12 →** 大家好，今天我们来讲…
**00:01:45 →** 首先我们看第一个知识点…
**00:05:23 →** 这个地方要注意…
**00:12:08 →** 总结一下今天的重点…

故障保护

  • 超时：30s + 每段音频时长 × 2s
  • 重试：最多 3 次，指数退避（5s→15s→45s）
  • 空结果：跳过当前段，不中断整任务
  • 部分成功：每段完成后立即存 .part 文件
  • 模型：TeleAI/TeleSpeechASR（支持时间戳输出）
